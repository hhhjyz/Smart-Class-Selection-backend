# STSS 统一网关接入指南（给各业务组）

> 网关 = 一个 nginx（仓库 `STSS-gateway`）。它统一做：**鉴权 + 注入身份头 + CORS + 路由 + 静态托管**。
> 各业务组后端**只认网关注入的身份头，自己不解 JWT、不做 CORS**。本文是各组把自己服务挂到网关后面的操作说明。

---

## 1. 它怎么工作（一句话版）

```
浏览器 ──(Authorization: Bearer <用户JWT>)──▶ 网关
   网关 auth_request ──▶ auth_service POST /api/v1/internal/verify  (网关用自己的 service token)
   auth_service 验完 JWT ──▶ 返回响应头 X-User-Id / X-User-Role / X-User-Permissions
   网关把这几个头【注入】到原请求，再转发给你的服务
你的服务 ──读这几个头即可拿到身份（不要再验 JWT）
```

验不过（401）时请求**根本到不了你的服务**；所以你的服务可以默认"来的请求都已鉴权"。

---

## 2. 各组后端必须遵守的契约（重点）

1. **身份只从这三个请求头读**（网关注入，可信）：
   - `X-User-Id`：用户 id
   - `X-User-Role`：角色，**大写** `ADMIN` / `TEACHER` / `STUDENT`
   - `X-User-Permissions`：权限码，**逗号分隔**字符串（如 `data-provision:read,user:read`）
   - 另有 `X-Request-ID` 用于日志串联，建议透传。
2. **不要自己解 JWT、不要自己校验签名**——网关已经做了。
3. **不要自己加 CORS 头**——网关统一加了，重复会冲突。
4. 缺头 = 非法请求，直接拒（正常流量网关保证有头）。
5. **内部接口**（只给别的服务调、不对公网）放在 `/api/v1/internal/*` 路径下——网关对外部访问该前缀一律 **403**。

> 各语言读法都一样，就是读 HTTP header。FastAPI 例：`Header(alias="X-User-Id")`。

---

## 3. 怎么把你的服务挂上去

给网关维护者（本组）一份信息 + 一段 location 片段即可：

**你要提供：**
- 服务的容器名/地址端口（如 `schedule-service:8000`）
- 你对外的**路由前缀**（建议 `/api/v1/<你的域>/`，如 `/api/v1/schedule/`）
- 特殊需求：大 body 上传？SSE/长连接？需要单独限流？

**location 片段模板**（照抄改三处：`<svc>`、`<upstream>`、`<prefix>`）：

```nginx
upstream <svc>_upstream {
    server <容器名>:<端口>;
    keepalive 32;
}

location ~ ^<prefix> {
    auth_request /internal/auth/verify;     # ← 关键：和 auth/info 同一套鉴权
    proxy_pass http://<svc>_upstream;
    # 全局 proxy_set_header 已注入 X-User-Id/Role/Permissions，无需在此重复
    # 如有 SSE/长连接：proxy_buffering off; proxy_read_timeout 3600s; proxy_http_version 1.1;
}
```

> 前提（网关主配置里已有，别重复定义）：`location = /internal/auth/verify`、全局 `proxy_set_header X-User-*`。
> 参考实物：本组的 `deploy/gateway-course-selection.conf`。

---

## 4. 服务间调用（你的服务去调别人）

跨组调用走 **service token**（不是用户 JWT）：
1. 用本组的 `client_id` / `client_secret` 调 `POST /api/v1/auth/sys/login` 换 service token；
2. 调用时带 `Authorization: Bearer <service token>`；
3. 典型场景：调 A 组 `info_service` 的 `/api/v1/info/data-provision/*`（培养方案、教师/学生名册等，专为下游准备，service token 鉴权）。

> 最新 A 组 `group1-base` 中，service token 权限来自 Auth DB 的 `SERVICE` 角色，不再读取
> `SERVICE_CLIENT_*_SCOPE`。选课服务依赖的最低权限是
> `data-provision:read`、`course:read`、`offering:read`、`schedule:read`、`classroom:read`。
> 如果使用旧的 `auth_data` 卷，需要重新 seed 或手动补齐 `SERVICE` 角色权限。

> 注意 service-to-service **走内网直连对方服务**，不经过网关那层用户鉴权。

---

## 5. 不需要 token 的路由（网关已豁免）

`/auth/login`、`/auth/sys/login`、`/auth/refresh`、`/auth/public-key` —— 登录/换票本身不要鉴权，其余一律要。

---

## 6. 各组自查清单

- [ ] 服务读 `X-User-Id` / `X-User-Role`(大写) / `X-User-Permissions`(逗号分隔)，不解 JWT
- [ ] 没自己加 CORS 头
- [ ] 内部接口在 `/api/v1/internal/*` 下
- [ ] 给了网关维护者：容器地址 + 路由前缀 + 特殊需求
- [ ] 跨组调用用 service token，不是转发用户 token
